01 — App Router (Your World)
These are the ones that matter most for your portfolio

These snippets are built for the App Router — the architecture your portfolio uses. Learn these first, use them daily. The narootlayout one alone will save you a lot of typing.

narootlayout→
Root Layout
The shell that wraps your entire app. Lives at app/layout.tsx. This is where your Navbar, Footer and global providers go. You only write this once.
Your portfolio app/layout.tsx — written once, wraps everything 
export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        <main>{children}</main>
      </body>
    </html>
  );
}
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        <main>{children}</main>
      </body>
    </html>
  );
}
napage→
App Router Page
Your go-to snippet for creating any new page in the App Router. Use this for app/page.tsx, app/about/page.tsx, app/projects/page.tsx — every page file.
Your portfolio JS + TS Use for every new page file you create 
export default function Page() {
  return <div></div>;
}
naloading→
Loading UI
A special file Next.js automatically shows while a page is loading. Create a loading.tsx next to any page.tsx and it becomes the skeleton/spinner for that route. Free streaming with zero extra code.
Your portfolio JS + TS Add loading.tsx next to any page — instant loading state 
export default function Loading() {
  return <div>Loading...</div>;
}
naerror→
Error Boundary
A special file Next.js shows if a page crashes. Must be a Client Component because it uses browser-side error catching. The reset function lets the user try again without a full page reload.
Your portfolio Create error.tsx next to any page — automatic error handling 
"use client";
import { useEffect } from "react";

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    console.error(error);
  }, [error]);

  return (
    <div>
      <h2>Something went wrong!</h2>
      <button onClick={reset}>Try again</button>
    </div>
  );
}
"use client";

    </div>
  );
}
02 — Component Shells (Use These Daily)
Your two go-to component starters

These are the function declaration style you said you prefer. Use nfe→ when you want to name the component and export it at the bottom. Use nf→ for a quick inline export. Both are identical in behavior.

nfe→
Named Function — Export at End
Function declaration with the export default at the bottom. Good for components where you want to see the name clearly separated from the export.
Your portfolio JS + TS Use for most components — rename FileName to your component name 
function FileName() {
  return <div>CONTENT</div>;
}

export default FileName;
nf→
Anonymous Function — Inline Export
Quickest possible component shell. The export default is inline so there's no separate export line at the bottom. Fine for simple, small components.
Your portfolio JS + TS Use for quick small components 
export default function () {
  return <div>CONTENT</div>;
}
03 — Route Handlers (API Endpoints)
What route handlers are

These replace the old pages/api/ pattern. You create a route.ts file inside any folder in app/ and export functions named GET, POST, etc. You'll use these for your contact form — the form submits a POST request, your route handler receives it and sends the email.

nrget→
Route Handler — GET
Handles GET requests to that route. Use when you need to read or fetch data from an endpoint. Lives in app/any-folder/route.ts.
Your portfolio app/api/contact/route.ts — for fetching data 
export async function GET(request: Request) {}
export async function GET(request) {}
nrpost→
Route Handler — POST
Handles POST requests. This is the one you'll use for your contact form — the form sends a POST with the message data, this function receives it.
Your portfolio Contact form submission — your main use case 
export async function POST(request: Request) {}
export async function POST(request) {}
04 — Imports & Usage
These save the most keystrokes

You'll type these constantly. nil→ and nul→ especially — Next.js Link and Image are used on nearly every page.

nil→
Import Next.js Link
Imports the Link component from next/link. Always use this instead of a plain <a> tag for internal navigation — it prefetches pages and makes the site feel instant.
Your portfolio JS + TS Any time you link to another page on your site 
import Link from "next/link";
nul→
Next.js Image Component
The Next.js Image component with src, width, height and alt. Always use this over a plain <img> tag — it automatically optimises images, lazy loads them, and prevents layout shift.
Your portfolio JS + TS Profile photo, project screenshots — any image on the site 
<Image src="path" width="width" height="height" alt="alt" />
niur→
Import useRouter
Imports the useRouter hook. Lets you programmatically navigate between pages — useful for redirecting after form submission or reading the current route in a component.
Your portfolio JS + TS When you need to navigate programmatically — e.g. after form submit 
import { useRouter } from "next/router";
05 — Good to Know
Not needed right now — but worth understanding

These won't come up in your day-to-day portfolio build but knowing what they do makes you a more complete Next.js developer. Read through them once, come back when you need them.

nalayout→
Nested Layout
A layout that only wraps a specific section of the app. If you add a layout.tsx inside app/projects/, everything under /projects shares that layout. Useful if your projects section needs a sidebar or different header.
Good to know app/projects/layout.tsx — section-specific layout 
export default function Layout({ children }: { children: React.ReactNode }) {
  return <section>{children}</section>;
}
export default function Layout({ children }) {
  return <section>{children}</section>;
}
natemplate→
Template
Like a layout but it re-renders on every navigation. Layouts persist and don't re-render — templates do. Use a template when you need animations to replay every time a user visits a route.
Good to know When you need page enter animations to replay on every visit 
export default function Template({ children }: { children: React.ReactNode }) {
  return <div>{children}</div>;
}
export default function Template({ children }) {
  return <div>{children}</div>;
}
nnotfound→
404 Not Found Page
A custom 404 page. Name the file not-found.tsx and Next.js automatically serves it when a route doesn't exist. Great opportunity to add personality — make yours match your site's aesthetic.
Good to know app/not-found.tsx — shown on any 404 
export default function NotFound() {
  return (
    <div>
      <h1>Page Not Found</h1>
    </div>
  );
}
06 — Deprecated — Don't Use
⚠ Old patterns — skip these entirely

These snippets generate code from the old Pages Router era or use class-based patterns that are no longer recommended. The extension still includes them for backwards compatibility with older projects. For your portfolio — ignore them completely.

ntsfwserver→
TS Function + getServerSideProps
Pages Router pattern — getServerSideProps doesn't exist in the App Router. Don't use.
DeprecatedPages Router only — not for your project
// ❌ Don't use — Pages Router pattern only
import { GetServerSideProps } from "next";

const FileName = () => {
  return <div>CONTENT</div>;
};

export const getServerSideProps: GetServerSideProps = async (ctx) => {
  return { props: { data: null } };
};

export default FileName;
ntscdocument→
TS Custom Document (Class-based)
The old class-based _document pattern. Replaced by the root layout in the App Router. Don't use.
DeprecatedUse narootlayout instead
// ❌ Don't use — replaced by narootlayout in App Router
import Document, { Html, Head, Main, NextScript } from "next/document";

class MyDocument extends Document {
  static async getInitialProps(ctx) { ... }
  render() { ... }
}

export default MyDocument;
