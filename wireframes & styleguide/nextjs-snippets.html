⚠ App Router vs Pages Router

These snippets use the Pages Router pattern (getStaticProps, getServerSideProps, _app, _document).
Your portfolio uses the App Router — so most of these won't apply to your day-to-day work.
The snippets still useful for you right now: impn→, np→ / tsnp→ as a quick component shell, and spth→ / tsspth→ for understanding the [slug] pattern conceptually.

01 — Page Components
What these do

Generate the basic shell of a Next.js page component so you don't have to type the boilerplate from scratch. Just type the prefix, hit Tab, and rename FileName to whatever your page is called.

np→
Basic Next.js Page
Generates a simple functional component with a default export. Starting point for any new page.
JavaScript Use when: creating a new .jsx page from scratch 
const FileName = () => {
  return <div></div>
}

export default FileName
tsnp→
Typed Next.js Page
Same as np but typed with NextPage and an empty Props interface. Use this in your portfolio since you're using TypeScript.
TypeScript Use when: creating a new .tsx page — your default 
import { NextPage } from 'next'

interface Props {}

const FileName: NextPage<Props> = () => {
  return <div></div>
}

export default FileName
02 — Static Data (Build Time)
What getStaticProps does

Runs at build time — fetches data once when you deploy, not on every visit. Perfect for content that doesn't change often like your projects list or journal posts. Faster than server-side rendering because the HTML is pre-built.

sp→
getStaticProps — function only
Just the function — use this when you want to add static props to a page you've already written.
JavaScript Use when: adding static data fetching to an existing page 
export const getStaticProps = async (context) => {
  return {
    props: {},
  }
}
tssp→
Typed getStaticProps — function only
Same as sp but with the GetStaticProps type from Next.js. TypeScript will catch any type mismatches in what you return.
TypeScript Use when: adding typed static props to an existing .tsx page 
export const getStaticProps: GetStaticProps = async (context) => {
  return {
    props: {},
  }
}
npsp→
Page + getStaticProps together
Full page component plus the getStaticProps function in one go. Use this when you know from the start your page needs build-time data.
JavaScript Use when: starting a data-driven page from scratch 
const FileName = () => {
  return <div></div>
}

export const getStaticProps = async (context) => {
  return {
    props: {},
  }
}

export default FileName
tsnpsp→
Typed Page + getStaticProps together
The TypeScript version of npsp. Typed page component plus typed static props. This is what you'll use for your projects and journal pages.
TypeScript Use when: starting a typed data-driven page — relevant for /projects 
import { GetStaticProps, NextPage } from 'next'

interface Props {}

const FileName: NextPage<Props> = () => {
  return <div></div>
}

export const getStaticProps: GetStaticProps = async (context) => {
  return {
    props: {},
  }
}

export default FileName
03 — Dynamic Routes
What getStaticPaths does

Required whenever you have a dynamic route like /projects/[slug] or /journal/[slug]. It tells Next.js which slugs actually exist so it can pre-build those pages. You return an array of all valid paths — Next.js builds a page for each one.

spth→
getStaticPaths
Generates the function that defines which dynamic routes exist. The params object keys must match your folder name — so [slug] means params: { slug: 'your-value' }.
JavaScript Use when: building /projects/[slug] or /journal/[slug] 
export const getStaticPaths = async () => {
  return {
    paths: [{ params: {} }],
    fallback: true,
  }
}
tsspth→
Typed getStaticPaths
Same as spth but with the GetStaticPaths type applied. TypeScript will enforce the correct return shape.
TypeScript Use when: building typed dynamic routes — your main use case 
export const getStaticPaths: GetStaticPaths = async () => {
  return {
    paths: [{ params: {} }],
    fallback: true,
  }
}
04 — Server-Side Rendering
What getServerSideProps does

Runs on the server on every single request — fetches fresh data each time someone visits the page. Slower than static but useful for data that changes constantly. For your portfolio you probably won't need this — your projects and journal content doesn't change per-visitor. But good to know it exists.

ssp→
getServerSideProps — function only
Just the SSR function. Add this to a page when you need data that's different on every request.
JavaScript Use when: the page needs fresh data on every visit 
export const getServerSideProps = async (context) => {
  return {
    props: {},
  }
}
tsssp→
Typed getServerSideProps — function only
The typed version. Note the extra 's' — tsssp not tsssp. Three s's total. Easy to mix up.
TypeScript Use when: typed SSR on an existing page 
export const getServerSideProps: GetServerSideProps = async (context) => {
  return {
    props: {},
  }
}
npssp→
Page + getServerSideProps together
Full page component plus SSR function. Same as npsp but server-rendered instead of static.
JavaScript Use when: starting a server-rendered page from scratch 
const FileName = () => {

}

export const getServerSideProps = async (context) => {
  return {
    props: {},
  }
}

export default FileName

05 — App & Document
What _app and _document are

_app wraps every page — use it for global providers, layouts, and styles that apply everywhere (like your Navbar and theme toggle).
_document is the HTML shell — the actual <html>, <head>, <body> tags. You only touch this if you need to add something to the HTML structure itself, like a custom font link or a lang attribute.

capp→
Custom _app.js
The global app wrapper. Wrap Component with your providers here — theme context, Framer Motion AnimatePresence for page transitions, global layout.
JavaScript Use when: setting up _app.js for global layout and providers 
const MyApp = ({ Component, pageProps }) => {
  return <Component {...pageProps} />
}

export default MyApp
tscapp→
Typed Custom _app.tsx
Same as capp but typed with AppProps. This is the version you'll use since your portfolio is TypeScript.
TypeScript Use when: setting up your typed _app.tsx — your version 
import { AppProps } from 'next/app'

const MyApp = ({ Component, pageProps }: AppProps) => {
  return <Component {...pageProps} />
}

export default MyApp
cdoc→
Custom _document with getInitialProps
Minimal _document setup. Only use this if you actually need to customise the HTML document — most projects never touch it.
JavaScript Use when: you need to modify the HTML document shell 
import Document from 'next/document'

class MyDocument extends Document {
  static async getInitialProps(ctx) {
    const initialProps = await Document.getInitialProps(ctx)
    return initialProps
  }
}

export default MyDocument
cdocdom→
Custom _document with full DOM tree
Full control — exposes the Html, Head, Main and NextScript elements. Use this when you need to add something directly to the HTML tag like lang attribute or a meta tag that must be in the document head.
JavaScript Use when: you need full control over the HTML structure 
import Document, { Html, Head, Main, NextScript } from 'next/document'

class MyDocument extends Document {
  static async getInitialProps(ctx) {
    const initialProps = await Document.getInitialProps(ctx)
    return { ...initialProps }
  }

  render() {
    return (
      <Html>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </Html>
    )
  }
}

export default MyDocument
